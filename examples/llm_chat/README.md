# Cute LLM chat — Ollama-backed local chat client

A minimal native LLM chat written in [Cute](https://github.com/i2y/cute).
Talks to a local [Ollama](https://ollama.ai/) daemon over `/api/chat`,
streams the response token-by-token via NDJSON, and renders the
conversation in QML with markdown + per-bubble syntax-highlighted
code blocks.

Scope is intentionally small: single conversation, one model, no
plugins, no RAG, no settings UI — it's a showcase of how the
Cute-on-Qt streaming-chat *primitives* fit together (HTTP client,
NDJSON parser, ModelList feed, QSyntaxHighlighter on a QML
TextEdit), not a feature-complete Ollama frontend.

## Bundle size

On macOS arm64 (Apple Silicon), Qt 6.11 from Homebrew:

- Cute-emitted binary: **180 KB**
- `cute build`-only `.app` (Qt linked dynamically): **188 KB**
- Self-contained `.app` after `macdeployqt` (Qt6 + QtQuick + ICU
  bundled in): **132 MB**, **42 MB** as `.tar.zst`

The deployed bundle is dominated by `libicudata.78.dylib` (32 MB,
Qt's bundled Unicode tables) plus the QtQuick / QtQml frameworks.
A Qt build with `-icu-config minimal` would cut that roughly in
half.

## Run

Prerequisite: Ollama installed and serving locally.

```sh
brew install ollama        # or your platform's equivalent
ollama serve &
ollama pull qwen2.5-coder:7b    # ~5 GB; or `qwen2.5:0.5b` for ~400 MB
```

Build + launch the QML chat:

```sh
cute build examples/llm_chat/llm_chat.cute
open llm_chat.app                    # macOS
# ./llm_chat                         # Linux (build CMake target directly)
```

Distribute as a self-contained bundle (Qt6 frameworks copied in,
no Homebrew dependency at run-time):

```sh
./examples/llm_chat/deploy.sh        # → llm_chat-deployed.app + .tar.zst
```

The script runs Qt's `macdeployqt` with the right `-libpath` flags
for Homebrew's keg-only layout, then suppresses the noisy
`Cannot resolve rpath "@rpath/Qt…"` warnings — they're transient
QML-module dependencies (QtVirtualKeyboard, QtPdf, Qt3D, …) that
the chat never touches at runtime, so the bundle works without
them.

Or run the CLI smoke that mirrors the chat's network path:

```sh
cute build examples/llm_chat/ollama_smoke.cute
./ollama_smoke "say hi in 4 words"
# → POST http://localhost:11434/api/chat (model=qwen2.5-coder:7b)
# → prompt: say hi in 4 words
# → ---
# → [token 1] Hello
# → [token 2]  from
# → [token 3]  the
# → [token 4]  digital
# → [token 5]  realm
# → [token 6] .
# → ---
# → [done] 6 tokens
```

`OLLAMA_HOST` and `OLLAMA_MODEL` env vars override the defaults.

## What's in this directory

- **`llm_chat.cute`** — the QML app: `class Message`, `class Chat`,
  `view Main`. Single file, ~330 LOC including comments.
- **`ollama_smoke.cute`** — CLI that exercises the same POST + streaming
  NDJSON parse. Useful for verifying Ollama connectivity without
  bringing up the GUI.
- **`ndjson_stream.cute`** — reusable `NdjsonStream` class that turns a
  `QNetworkReply` into a `received(QJsonObject)` signal stream. The
  showcase chat inlines the parsing for clarity, but this class is the
  refactor target if a second NDJSON consumer ever wants in.

## Architecture notes

The chat client is structured as:

1. `class Message { prop role, prop content }` — reactive QObject row
   item, displayed in the message bubble.
2. `class Chat { prop messages : ModelList<Message>, ... }` — single
   instance owned by the QML `view`. Holds the conversation, the
   in-flight reply, the streaming buffer, and the running history JSON.
   Method `send(text)` does the POST + connects readyRead/finished.
3. `view Main { ApplicationWindow { ... } }` — pure-Cute QML that binds
   to `Chat`'s reactive props: ListView feed, "currently typing" bubble
   (visible while `isStreaming`), TextField + Send button.

The streaming path:

```
TextField onAccepted → Chat.send(text)
  → manager.post(req, body)
  → reply.readyRead → drain NDJSON lines from buffer
                     → for each obj: append `obj.message.content`
                       to Chat.currentDraft (QML rebinds reactively)
  → reply.finished  → promote currentDraft into a permanent Message,
                       reset, mirror into history JSON for the next turn
```

No QML hand-written: every `.qml` view in this app is generated by the
Cute compiler from the `view Main { ... }` block. The window chrome,
fonts, scroll behavior, and rendering pipeline are all native Qt
QtQuick.

## Foundation work landed alongside

This app uncovered (and motivated landing) several Cute language /
binding improvements that have value beyond the chat itself:

- **`QNetworkReply` streaming surface** — `readyRead`, `bytesAvailable`,
  `read`/`readLine`, `errorString` are now exposed via the
  `extra_signals` / `extra_methods` typesystem mechanism. Without this,
  no Cute app could observe HTTP chunks before `finished`.
- **`QProcessEnvironment` binding** — read OS env vars like
  `OLLAMA_HOST` / `ANTHROPIC_API_KEY` without going through QSettings.
- **`QByteArray` buffer surface** — `append`, `indexOf`, `mid`, `left`,
  etc. for streaming consumers that need to split-on-delimiter.
- **Lambda capture mode in class methods** — Cute now emits
  `[=, this] mutable` for signal-connect lambdas inside class / struct
  methods (previously `[&]`, which left dangling references to the
  enclosing method's parameters once the method returned).
- **Signal param convention** — header and source-side signal
  declarations now agree on by-value parameter shape, so a `signal
  X(msg: String)` no longer fails to link.
- **`detect_build_extras`** — QML and CLI apps that pull in
  `QNetworkAccessManager` automatically get `Qt6::Network` linked, no
  cute.toml entry required.
- **`ModelList<T>` element accessor typing** — `messages.at(i).role`
  inside a class method now lowers to `messages()->at(i)->role()`,
  letting Cute code iterate a model-list with the same shape as the
  C++ `cute::ModelList<T>::at(int) -> T*` API. The `messages : ModelList<Message>`
  prop drives both the QML feed and the JSON-history serializer in
  `Chat::buildBody`.
- **Default QML Controls style → `Basic`** — `qml_app(Main)` now
  calls `QQuickStyle::setStyle("Basic")` when `QT_QUICK_CONTROLS_STYLE`
  isn't set in the environment, so user-customised
  `Button.background` / `TextField.background` work out of the box
  instead of being silently dropped by the macOS-native style.
